🌍 Proxy e Desbloqueio Regional

Como o OmniRoute contorna bloqueios geográficos (ex: unsupported_country_region_territory) com proxy de 3 níveis, TLS fingerprint spoofing e CLI fingerprint matching.

🚫 O Problema: Bloqueio Regional

Provedores como OpenAI/Codex bloqueiam acesso de certos países/regiões. Usuários recebem erros como:

unsupported_country_region_territory
Error: OAuth failed - your region is not supported
The API is not available in your country

Isso afeta especialmente desenvolvedores em países em desenvolvimento, impedindo acesso a ferramentas de IA essenciais.

🛡️ Solução do OmniRoute: 3 Níveis de Proxy

O OmniRoute configura proxy em 3 níveis hierárquicos, cada um com granularidade diferente:

Nível Escopo Badge Casos de Uso
1. Global Todo o tráfego 🟢 Verde VPN corporativa, proxy único para tudo
2. Por Provider Um provider específico 🟡 Amarelo OpenAI via proxy, DeepSeek direto
3. Por Connection/Key Uma conta/chave específica 🔵 Azul Conta X via proxy Brasil, Conta Y direta
💡 Visual no Dashboard: Cada conexão mostra badge colorido indicando qual nível de proxy está ativo + IP de saída. Configure em Dashboard → Providers → [Provider] → Connection → Proxy.

🔧 Configurando Proxy

Via Dashboard (Recomendado)

  1. Vá em Providers e clique no provider desejado
  2. Clique na conexão (conta) específica
  3. Na seção Proxy Config, preencha:
Proxy URL: http://user:pass@proxy.example.com:8080
# ou SOCKS5:
Proxy URL: socks5://user:pass@proxy.example.com:1080

✅ Suporta HTTP/HTTPS e SOCKS5

Via Variáveis de Ambiente (Global)

# Proxy HTTP/HTTPS global
export HTTP_PROXY=http://proxy.example.com:8080
export HTTPS_PROXY=http://proxy.example.com:8080

# Ou no .env do OmniRoute
HTTP_PROXY=http://proxy.example.com:8080
HTTPS_PROXY=http://proxy.example.com:8080

# Inicie
omniroute
⚠️ OAuth também passa pelo proxy: O fluxo OAuth (Claude Code, Codex, Copilot) é roteado pelo proxy configurado, resolvendo unsupported_country_region_territory durante autenticação.

🥷 TLS Fingerprint Spoofing (JA3/JA4)

Muitos provedores detectam bots/clientes não-nativos via TLS fingerprint (ordem de cipher suites, extensões, etc.). O OmniRoute usa wreq-js para imitar fingerprint de navegador real.

Como funciona

  • Intercepta handshake TLS de saída
  • Reordena cipher suites para combinar com Chrome/Firefox
  • Adiciona/remove extensões TLS para matching
  • Gera JA3/JA4 fingerprint indistinguível de navegador

Ativado automaticamente quando proxy está configurado. Funciona para qualquer provider.

✅ Resultado: Drasticamente reduz risco de flag de conta. O IP do proxy é preservado — você ganha stealth + IP masking simultaneamente.

🔏 CLI Fingerprint Matching

Provedores também detectam clientes via estrutura da requisição HTTP (ordem de headers, formatação JSON, campos extras). O OmniRoute reordena headers e body para matchar binários nativos:

CLI O que o OmniRoute Imita
Claude CodeHeader order, Anthropic-Version, beta headers, cache-control
Codex CLIOpenAI-Organization, User-Agent, request ID format
GitHub CopilotEditor-Version, Copilot-Integration-ID, VS Code fingerprint
OpenCodeCustom headers, streaming format, tool call structure
💡 Ativação: Automático quando você usa variáveis de ambiente nativas (ANTHROPIC_BASE_URL, OPENAI_BASE_URL). O proxy IP + TLS stealth + CLI fingerprint = proteção máxima.

🌐 Proxy SOCKS5

Suporte completo a SOCKS5 para roteamento avançado:

Exemplos de URL SOCKS5

# Básico
socks5://proxy.example.com:1080

# Com autenticação
socks5://user:pass@proxy.example.com:1080

# SOCKS5h (resolve DNS no proxy - recomendado para anonimato)
socks5h://user:pass@proxy.example.com:1080

Casos de Uso

  • Tor/onion routing
  • SSH tunnels dinâmicos
  • Proxies residenciais rotativos
  • WireGuard/VPN split tunneling

Configuração no Dashboard

Mesmo campo "Proxy URL" — apenas use esquema socks5:// ou socks5h://. Badge 🔵/🟡/🟢 aparece automaticamente.

☁️ Guia: OAuth Remoto (VPS/Cloud/Docker)

Rodar OmniRoute em servidor remoto requer configuração extra para OAuth funcionar:

Problema

OAuth precisa de redirect_uri acessível. Em VPS/Docker, localhost não funciona do navegador do usuário.

Solução: Manual URL Mode

  1. No Dashboard → Providers → [Provider] → Connection → OAuth Settings
  2. Ative "Manual URL Mode"
  3. Configure REDIRECT_URI=https://seu-dominio.com/callback
  4. Configure DNS + Nginx/Caddy proxy reverso para :20129 (dashboard port)
  5. OAuth flui: Browser → Nginx → OmniRoute Dashboard → Provider

Nginx Config Exemplo

server {
  listen 443 ssl;
  server_name seu-dominio.com;

  location / {
    proxy_pass http://localhost:20129;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
    
    # WebSocket support para A2A/MCP
    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection "upgrade";
  }
}

✅ Checklist de Desbloqueio

Siga em ordem até funcionar:

  1. Proxy HTTP/SOCKS5 funcional (teste com curl -x proxy http://ipinfo.io)
  2. Configure no nível correto (Global / Provider / Connection)
  3. Badge de proxy aparece no dashboard (🟢/🟡/🔵)
  4. Teste conexão do provider — deve mostrar IP do proxy
  5. Teste OAuth (Claude Code/Codex) — deve autenticar via proxy
  6. Faça uma request real — verifique IP de saída no dashboard

🔧 Troubleshooting

Configure Manual URL Mode no provider e use domínio real (não localhost). O OmniRoute usa window.location.origin automaticamente atrás de reverse proxy.

Verifique se o proxy aceita conexões do IP do servidor OmniRoute. Teste: curl -x http://proxy:port http://httpbin.org/ip do mesmo host.

Use socks5h:// (com 'h' no final) para resolver DNS no lado do proxy, não localmente.

Verifique: 1) TLS fingerprint está ativo (padrão com proxy), 2) CLI fingerprint matching (use vars ANTHROPIC_BASE_URL/OPENAI_BASE_URL), 3) Proxy IP não está em lista negra conhecida, 4) Não reutilize conta em múltiplos locais simultaneamente.
